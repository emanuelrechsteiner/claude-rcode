# Planning-Doc Convention Rule

> Planning docs (PLANNING.md, PLAN.md, ARCHITECTURE.md, SPEC.md, DESIGN.md) are versioned single-source-of-truth artifacts, not draft notebooks. Derived from IMP-036 (an example PLANNING.md with rewrite-heavy sessions: 4.4M chars across 229 edits at 80% rewrite rate, 2026-05). Always loaded.

## Scope

A "planning doc" is any file matching these filename patterns at any level of the repo:

- `PLANNING.md`, `PLAN.md`, `PLANS.md`
- `ARCHITECTURE.md`, `ARCH.md`
- `SPEC.md`, `SPECIFICATION.md`, `SPECS.md`
- `DESIGN.md`, `DESIGNS.md`
- Files under `docs/plans/`, `docs/specs/`, `docs/architecture/`

The convention applies even when these docs live inside subprojects.

## The Convention

**Planning docs are LIVE SPECS.** They describe the current intended state of the system and the work being done. They are not scratch space for in-progress thinking.

Operationally:

1. **Commit changes to a planning doc in dedicated commits.** Don't mix planning-doc edits with code edits in the same commit. The git history should let a reader follow how the plan evolved separately from how the code evolved.
2. **Don't refactor a planning doc in-place during code work.** If during implementation you discover the plan is wrong, finish the immediate code change, then update the plan in a separate commit explaining what changed and why.
3. **Edits, not rewrites.** A planning doc should grow by amendment (clarification, new sections, struck-through obsolete sections) rather than full rewrites. A rewrite signals "this was a draft, not a spec."

## Anti-Pattern: Draft-Notebook Mode

Symptoms that a planning doc is being treated as draft notebook:

- 200+ edits in a single session
- 80%+ of edits are rewrites (`old_string` empty or `Write` tool)
- File size oscillates by >20% within a session
- The same section is rewritten 3+ times in a day

These suggest the doc is being used as a thinking space rather than a spec. The git history loses signal: 200 rewrites look the same as a single thoughtful update from outside the session.

Concrete evidence basis: an example `PLANNING.md` accumulated 4.4M chars across 229 edits at an 80% rewrite rate within a single week (IMP-036, Phase X3 Finding #05).

## Recommendation: Draft → Replace Workflow

When you genuinely need to rework a planning doc heavily (>50% of content changing):

```
1. Copy the original to PLANNING-draft-YYYY-MM-DD.md (or PLAN-draft-*.md)
2. Iterate in the draft file — any number of rewrites, no commit hygiene needed
3. When the draft is stable, replace the original in ONE final commit:
   - Commit message: "planning: rework X (draft → PLANNING.md)"
   - Body: summarize what changed structurally and why
4. Delete the draft file in the same commit, or in an immediate follow-up
```

This preserves the "live spec" property of the original while giving you a scratch space when you need one.

## Session-End Warning

`session-end-check.sh` warns at the end of any session with ≥10 edits where ≥80% are rewrites. The warning cites this rule. Treat it as a prompt to use the draft → replace workflow next time, not as a hard error.

## What This Rule Is NOT

- Not a ban on editing planning docs — frequent small amendments are healthy
- Not a coverage gate or pre-commit blocker — it's social pressure via session-end surfacing
- Not applicable to scratch docs (BRAINSTORM.md, NOTES.md, TODO.md) which ARE explicitly drafts
- Not applicable to autogenerated docs (API references, schema dumps) — those follow their own update cadence
