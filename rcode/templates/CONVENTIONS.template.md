## Folder Structure

```
[project-root]/
├── src/                        # Source code
│   ├── app/                    # [Framework-specific app directory]
│   │   ├── (routes)/           # Route groups
│   │   ├── api/                # API routes
│   │   └── layout.tsx          # Root layout
│   ├── components/             # Reusable UI components
│   │   ├── ui/                 # Primitive UI components
│   │   └── [feature]/          # Feature-specific components
│   ├── lib/                    # Shared utilities and helpers
│   │   ├── utils.ts            # General utilities
│   │   ├── constants.ts        # App constants
│   │   └── types.ts            # Shared TypeScript types
│   ├── hooks/                  # Custom React hooks (if applicable)
│   ├── stores/                 # State management (if applicable)
│   ├── services/               # External service integrations
│   └── styles/                 # Global styles and theme
├── tests/                      # Test files (mirrors src/ structure)
├── public/                     # Static assets
├── docs/                       # Additional documentation
└── .rcode/                # Workflow state
```

### Rules

- **Components** go in `src/components/` — grouped by feature, not by type
- **Shared utilities** go in `src/lib/` — never in component directories
- **Types** shared across features go in `src/lib/types.ts` — feature-local types stay in feature directory
- **API routes** follow the route structure in `src/app/api/`
- **Tests** mirror the source structure in `tests/`

### Correct vs Incorrect

```
# CORRECT: Feature-grouped components
src/components/auth/LoginForm.tsx
src/components/auth/SignupForm.tsx
src/components/venues/VenueCard.tsx
src/components/venues/VenueList.tsx

# INCORRECT: Type-grouped components
src/components/forms/LoginForm.tsx
src/components/forms/SignupForm.tsx
src/components/cards/VenueCard.tsx
src/components/lists/VenueList.tsx
```

## File Naming

| Category | Convention | Example |
|----------|-----------|---------|
| Components | PascalCase | `VenueCard.tsx` |
| Hooks | camelCase with `use` prefix | `useVenueData.ts` |
| Utilities | camelCase | `formatDate.ts` |
| Constants | camelCase (file), UPPER_SNAKE (exports) | `constants.ts` → `MAX_RETRY_COUNT` |
| Types/Interfaces | PascalCase | `types.ts` → `interface Venue` |
| API Routes | kebab-case (directory) | `app/api/auth/login/route.ts` |
| Test Files | Same as source + `.test` | `VenueCard.test.tsx` |
| Style Files | Same as component + `.module` | `VenueCard.module.css` (if CSS modules) |

## State Management Patterns

### When to Use What

| State Type | Solution | Example |
|------------|----------|---------|
| UI State (local) | `useState` | Form inputs, toggles, open/closed |
| Server Data | [Data fetching library] | API responses, cached data |
| Global UI State | [State management library] | Theme, sidebar state, notifications |
| URL State | URL params / searchParams | Filters, pagination, active tab |
| Form State | [Form library or useState] | Complex form with validation |

### Rules

- **Never duplicate server state in client state** — use the data fetching library's cache
- **URL state over client state** for filterable/shareable UI
- **Local state by default** — only lift to global when truly needed

## Error Handling Patterns

### API Route Errors

```typescript
// Standard error response format
{
  error: {
    code: 'VALIDATION_ERROR',     // Machine-readable code
    message: 'Invalid email',      // Human-readable message
    details: { field: 'email' }    // Optional additional context
  }
}

// Standard HTTP status codes
// 400 — Validation error (client mistake)
// 401 — Not authenticated
// 403 — Not authorized (authenticated but no permission)
// 404 — Resource not found
// 409 — Conflict (duplicate, stale data)
// 500 — Server error (our mistake)
```

## Testing Patterns

### Test Structure

```typescript
describe('[ComponentName]', () => {
  // Setup
  beforeEach(() => { /* common setup */ })

  // Happy path first
  it('renders correctly with required props', () => { })

  // Interaction tests
  it('handles user click on submit button', () => { })

  // Edge cases
  it('shows error when form is submitted empty', () => { })

  // Accessibility
  it('has correct ARIA labels', () => { })
})
```

### What to Test

| Always Test | Sometimes Test | Never Test |
|-------------|---------------|------------|
| Business logic | CSS class names | Third-party library internals |
| User interactions | Exact error wording | Framework behavior |
| Error states | Animation timing | Implementation details |
| Accessibility | Responsive layout | Private functions directly |
| API contracts | Performance | Snapshot tests (avoid) |

## API Patterns

### Request/Response Format

```typescript
// Successful response
{ data: T, meta?: { page: number, total: number } }

// Error response
{ error: { code: string, message: string, details?: unknown } }
```

### Naming Conventions

| Operation | HTTP Method | Route Pattern | Example |
|-----------|-------------|---------------|---------|
| List | GET | `/api/[resource]` | `GET /api/venues` |
| Get One | GET | `/api/[resource]/[id]` | `GET /api/venues/123` |
| Create | POST | `/api/[resource]` | `POST /api/venues` |
| Update | PATCH | `/api/[resource]/[id]` | `PATCH /api/venues/123` |
| Delete | DELETE | `/api/[resource]/[id]` | `DELETE /api/venues/123` |
